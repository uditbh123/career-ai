# ADR-003: Use a Monorepo Architecture

**Status:** Accepted

**Date:** 2026-07-10

**Decision Makers:** Career AI Core Team

## Context

Career AI consists of multiple components that evolve together:

* Backend API
* Frontend
* Documentation
* Infrastructure
* Docker configuration
* CI/CD
* Future AI modules

A repository strategy must support long-term growth while remaining approachable for contributors.

---

# Problem Statement

Determine whether the project should be maintained as a single repository or divided into multiple repositories.

---

# Architectural Drivers

* Developer experience
* Simplicity
* Documentation
* Version consistency
* Ease of onboarding
* Maintainability

---

# Alternatives Considered

## Monorepo

### Advantages

* Single source of truth
* Easier onboarding
* Unified documentation
* Shared versioning
* Simpler CI/CD
* Easier refactoring
* Atomic commits across backend and frontend

### Disadvantages

* Repository grows larger over time
* CI pipelines may become more complex

---

## Separate Repositories

### Advantages

* Independent deployments
* Smaller repositories
* Team autonomy

### Disadvantages

* Version synchronization
* Duplicate documentation
* More difficult onboarding
* Cross-repository pull requests

---

## Microservices

### Advantages

* Independent scaling
* Technology flexibility
* Fault isolation

### Disadvantages

* Operational complexity
* Service discovery
* Distributed debugging
* Network overhead
* Excessive complexity for current project size

---

# Decision

Career AI will use a monorepo architecture.

---

# Rationale

The backend, frontend, infrastructure, and documentation evolve together.

A monorepo simplifies development by allowing architectural changes, documentation updates, infrastructure changes, and code modifications to be committed together.

Given the current project scope, the operational complexity of multiple repositories or microservices is not justified.

---

# Consequences

Positive:

* Easier maintenance
* Consistent documentation
* Unified releases
* Simpler contributor experience

Negative:

* Larger repository over time
* Shared CI pipeline

---

# Risks

* Repository could become disorganized.
* Team boundaries may become unclear.
* Build times may increase as the project grows.

---

# Risk Mitigation

* Maintain strict folder structure.
* Keep modules independent.
* Introduce workspace tooling as needed.
* Monitor CI performance.

---

# Migration Strategy

If the project grows substantially:

1. Separate deployable services first.
2. Preserve shared interfaces.
3. Extract repositories only when operational benefits outweigh added complexity.

The initial monorepo should not prevent future decomposition.

---

# Review Criteria

Revisit if:

* Multiple independent teams maintain the project.
* Independent deployment cycles become necessary.
* Repository size significantly impacts productivity.

---

# Decision Outcome

Accepted.

A monorepo provides the simplest and most maintainable architecture for the current stage of Career AI while preserving a clear migration path should the project outgrow this approach.
